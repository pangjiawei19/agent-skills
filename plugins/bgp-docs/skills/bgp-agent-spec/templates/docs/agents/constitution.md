# AI Agent 开发宪法

本文件定义了本项目不可动摇的核心开发原则。所有 AI Agent 在进行技术规划和代码实现时，必须无条件遵循。

---

## 第一条：简单性原则 (Simplicity First)

**核心：** 遵循"少即是多"哲学。绝不进行不必要的抽象，绝不引入非必需的依赖。

- **1.1:** 只实现 `spec.md` 中明确要求的功能。
- **1.2:** 简单的函数和数据结构优于复杂的接口和继承体系。
- **1.3:** 函数尽量小，尽量可以复用，尽量不写重复代码。
- **1.4:** 注意模块设计，尽量使用设计模式。

---

## 第一条附则：术语一致性原则 (Terminology Consistency)

**核心：** 同一概念在整个项目中只使用一个英文术语，从类型定义到 API 路径到前端组件，全链路统一。

- **1.5:** 一旦确定了某个概念的英文命名，所有层（数据库字段映射、API 路径、请求/响应字段、前端组件名）必须使用同一术语，禁止混用近义词。
- **1.6:** 示例：标的统一用 `instrument`，不混用 `security`；策略统一用 `strategy`，不混用 `policy`。

---

## 第二条：测试先行铁律 (Test-First Imperative) - 不可协商

**核心：** 测试是开发的前置条件，不是事后验证；新功能或 Bug 修复必须从一个失败的测试开始。

- **2.1:** 严格遵循"Red-Green-Refactor"循环。
- **2.2:** 单元测试必须覆盖业务函数的纯逻辑、边界值、错误分支。集成测试通过不能作为省略单元测试的理由。
- **2.3:** 集成测试必须使用真实依赖（数据库、HTTP 客户端、外部进程），不可用 mock 替代真实依赖。例外：第三方付费 API、不可控的时间/随机源。

---

## 第三条：明确性原则 (Clarity and Explicitness)

**核心：** 代码的首要目的是让人类易于理解。

- **3.1:** 解释代码的时候，说人话，别拽专业术语。最好有图（Mermaid 风格）。
- **3.2:** 改动或者解释代码前，必须先看所有相关代码，不能偷懒。
- **3.3:** 改动代码前，要做最小化修改，尽量不修改到其他模块的代码。
- **3.4:** 改动代码后，先自行 review 一遍，发现可能的语法及逻辑问题。

---

## 治理 (Governance)

本宪法具有最高优先级，其效力高于任何规则或单次会话中的指令。

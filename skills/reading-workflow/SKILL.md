# Reading Workflow — 本地读书全流程管理

## 触发条件

**严格匹配以下 4 个 shortcut，其他口语化表达不触发：**

| Shortcut | 动作 |
|----------|------|
| `+start <书名 \| URL \| 书名 + URL>` | 开始读：可选抓取链接元信息 + 启动问询 + 创建文件 |
| `+log <书名> <进度文本>` | 日常记录：更新进度表 / 章节笔记 / 感想 |
| `+done <书名>` | 标记读完：更新 meta 状态，主动问是否总结 |
| `+summary <书名>` | 生成 / 重写读后总结段（在原文件内追加或覆盖） |

书名可带或不带 `《》`，按 [Sanitize 规则](#书名主键规则sanitize) 统一处理。参数里含 `https?://` 就走 URL 抓取分支。

---

## 单文件原则

**一本书 = 一个文件 `reading/{key}.md`**。进度、笔记、感想、总结全部维护在同一份文件里。`+summary` 不另建文件，而是在原文件的 `## 📖 读后总结` 段落里追加 / 覆盖内容。

---

## 书籍类型

决定 `+summary` 时使用哪个段落模板：

| 类型 | 代号 | 适用场景 | 总结段模板 |
|------|------|----------|------------|
| 小说 / 虚构 | `fiction` | 小说、剧本、传记类虚构 | `reading-summary-fiction.md` |
| 技术 / 工具书 | `technical` | 编程、架构、工程实践 | `reading-summary-technical.md` |
| 通用 / 非虚构 | `general` | 商业、社科、心理、历史、人文 | `reading-summary-general.md` |

### 类型识别规则

**优先级：用户显式 > 关键词推断 > 主动询问**

1. **用户显式指定**：如 `+start 技术书《xxx》`
2. **关键词推断**：
   - fiction：小说、推理、科幻、奇幻、剧本、散文
   - technical：编程、架构、源码、框架、算法、系统设计、具体技术名（Kubernetes、Rust 等）
   - general：商业、管理、心理、经济、历史、传记（非虚构）、社科
3. **不确定时主动询问**：给出三个选项让用户选

## 书名主键规则（Sanitize）

用于构建文件名 `reading/{key}.md`。

**伪代码**：

```
key = 书名
key = strip(key, "《》「」『』\"\"''")    # 去引号 / 书名号
key = replace(key, /[：:\/\\]/, "-")       # 中英文冒号、斜杠 → 连字符
key = remove(key, /[*?\"<>|]/)              # 去文件系统非法字符
key = trim(key)                             # 去首尾空格
key = replace(key, /\s+/, "-")              # 内部连续空格 → 连字符
# 保留：中英文、数字、连字符、下划线、点
```

**示例**：

| 原始书名 | Key | 文件 |
|----------|-----|------|
| 《三体》 | `三体` | `reading/三体.md` |
| 《三体 II：黑暗森林》 | `三体-II-黑暗森林` | `reading/三体-II-黑暗森林.md` |
| 《The Pragmatic Programmer》 | `The-Pragmatic-Programmer` | `reading/The-Pragmatic-Programmer.md` |
| 《代码大全 2》 | `代码大全-2` | `reading/代码大全-2.md` |

## 核心流程

### 流程一：`+start` — 开始阅读

**三种调用形式**：
- `+start <书名>`：纯书名，跳过抓取直接启动问询
- `+start <URL>`：仅链接，AI 从页面提取书名
- `+start <书名> <URL>`：两者都给，书名以用户为准，URL 用于补充元信息

**分三段：URL 抓取（可选）→ 启动问询 → 创建文件。**

#### 第零段：URL 抓取（仅当参数含 URL 时）

```
1. 识别参数中的 URL（https?:// 开头）
2. 用 WebFetch 抓取页面，提取：
   书名 / 作者 / 出版社 / 出版年 / 页数 / ISBN / 豆瓣评分 / 内容简介 / 作者简介 / 读者标签
3. 抓取失败 → 提示用户手填关键字段（书名、作者、类型），不放弃流程
4. 抓取成功：
   a. 把内容简介用 AI 重写成 3-5 句要点（不照搬营销文案）
   b. 展示信息卡（书名 / 作者 / 出版社 / 页数 / 评分 / 简介摘要）让用户确认或纠正
   c. 类型按简介 + 读者标签预判（fiction / technical / general）
5. 进入第一段：基于简介给"为什么想读"的猜测时更精准
```

**支持的来源**：
- 显式支持：豆瓣读书（`book.douban.com`）
- 其他来源（微信读书、Goodreads、出版社官网等）：AI 尽力解析，不保证字段完整。字段缺失就留空，禁止编造。

#### 第一段：启动问询

```
1. 解析书名（去掉 +start 前缀，提取《》内或纯文本）
2. 识别书籍类型（显式 > 关键词推断 > 询问）
3. AI 主动给出 3-5 种"为什么想读"的可能性
   - 基于书名、已知作者 / 主题
   - 用"可能是因为..."的口吻，让用户挑选或补充
4. 一次性列出启动问询清单（共 7 项），让用户统一回复：
   ① 阅读工具（Kindle / 微信读书 / 纸质书 / ...）
   ② 阅读场景（通勤 / 睡前 / ...）
   ③ 每次阅读量（时间或章节 / 页数）
   ④ 每周阅读几次
   ⑤ 预估阅读周期（几周）
   ⑥ 为什么想读（从第 3 步选项里选 / 或自己说）
   ⑦ 读之前的想法 / 期待（可选）
5. 用户答不全 → 已答项填入，缺项留空（禁止编造）
```

#### 第二段：创建文件

```
1. 按 Sanitize 规则生成 key
2. 检查 reading/{key}.md 是否已存在
   - 存在 → 提示用户："已存在该书记录，是否覆盖？或用 +start <书名-v2> 区分"
   - 不存在 → 继续
3. 读取 references/reading-template.md
4. 填充：书名、作者、类型、开始日期（今天）、状态=🔄 进行中
5. 若有第零段抓取结果，填入「📚 书籍信息」段；否则该段标注"未抓取"或整段省略
6. 填充"🚀 阅读计划"区块（7 项启动问询答案）
7. 写入 reading/{key}.md
8. 回复："已创建《xxx》读书记录（类型：xxx，预计 N 周读完）"
```

**执行要点：**
- **启动问询不可跳过**：即使用户提供了 URL，问询的 7 项也要走完
- **动机推测要具体**：结合书名 / 作者 / 简介给出具体猜测，不要"可能对这个主题感兴趣"这种泛话
- **简介是参考，动机以用户为准**：豆瓣简介是营销文案，推测"为什么想读"时基于它给出选项，但不要引导用户接受书的卖点
- **清单一次列完**：不要一项项追问
- **禁止编造**：用户没答的、抓取没拿到的，一律留空

---

### 流程二：`+log <书名> <进度文本>` — 日常记录

```
1. 解析书名和进度信息
2. 按 Sanitize 规则定位 reading/{key}.md
   - 文件不存在 → 提示用户先用 +start 创建
3. 解析进度文本，分发到对应段落：
   - 章节 / 页数 / 百分比 → 新增一行到「📖 进度追踪」
   - 章节笔记内容 → 追加到「📝 章节笔记」
   - 感想 / 思考 → 追加到「💭 感想与思考」（带日期）
   - 金句 → 追加到「✨ 金句摘抄」
   - 疑问 → 追加到「❓ 疑问与待解」
4. 回复："已记录《xxx》进度：[简要说明更新了哪几段]"
```

**执行要点：**
- 进度文本是自由格式，AI 自行判断归属哪段（不确定就问）
- 日期统一用今天，除非用户显式指定
- 「💭 感想」是过程实时记录，与「📖 读后总结」里的"洞察"区分（后者是读完回看提炼）

---

### 流程三：`+done <书名>` — 标记读完

```
1. 按 Sanitize 规则定位 reading/{key}.md
2. 更新 meta 区块：
   - 状态：🔄 进行中 → ✅ 已完成
   - 完成日期：今天
3. 主动询问："要现在做总结吗？（+summary 触发，可以稍后再做）"
4. 回复："已标记《xxx》读完"
```

**执行要点：**
- `+done` 与 `+summary` 解耦，读完不一定立刻总结
- 状态切换不写入感想段，避免污染过程记录

---

### 流程四：`+summary <书名>` — 生成 / 重写总结

```
1. 按 Sanitize 规则定位 reading/{key}.md
2. 读取文件，提取：
   - meta（书名、作者、类型、起止日期）
   - 进度追踪、章节笔记、感想、金句、疑问 段全文
3. 根据 meta 里的「类型」选对应段落模板：
   - fiction → references/reading-summary-fiction.md
   - technical → references/reading-summary-technical.md
   - general → references/reading-summary-general.md
4. 基于已记录的笔记 / 感想，生成结构化总结内容
5. 写入文件的「📖 读后总结」段：
   - 该段为空 → 直接填入
   - 该段已有内容 → 询问用户「覆盖 / 追加新版本（标日期）/ 取消」
6. 回复："已写入《xxx》读后总结"
```

**执行要点：**
- 总结基于文件里已有的过程记录提炼，不要凭空发挥
- 用户笔记稀薄时（如只有进度没有感想），明确告诉用户「记录较少，总结会偏薄，是否补充几句再总结？」
- 总结段不应重复进度表 / 章节笔记 / 感想等已存在的内容，只放类型独有的提炼

## 注意事项

1. **禁止编造**：只记录用户亲口说的内容 / 抓取拿到的字段，不推断、不扩写
2. **启动问询不可跳**：`+start` 必须先问清 7 项再建文件（即使提供了 URL）
3. **单文件原则**：一本书一个 `.md`，进度和总结合并维护，不另建总结文件
4. **类型先定**：`+start` 时就确定类型，避免 `+summary` 时类型不清
5. **Sanitize 一致**：4 个 shortcut 都用同一规则定位文件
6. **`+done` 不自动总结**：主动询问但由用户决定
7. **过程 vs 提炼**：「💭 感想」是过程实时记录，「📖 读后总结」是读完回看的结构化提炼，两者不应混淆或重复
8. **简介是参考，动机以用户为准**：抓取来的简介只用于辅助 AI 给更具体的猜测，最终"为什么想读"以用户自述为准

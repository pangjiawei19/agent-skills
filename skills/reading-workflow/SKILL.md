# Reading Workflow - 读书全流程管理

## 触发条件

**当用户说以下任意触发词时触发本 skill：**
- "开始读《xxx》" / "开始读 xxx"
- "读了《xxx》" / "读了 xxx"
- "完成《xxx》阅读" / "完成 xxx 阅读"
- "总结《xxx》" / "总结 xxx"
- "聊聊《xxx》" / "聊聊 xxx"

---

## 书籍类型

支持三种类型，决定使用哪个总结模板：

| 类型 | 代号 | 适用场景 | 总结模板 |
|------|------|----------|----------|
| 小说/虚构 | `fiction` | 小说、剧本、传记类虚构 | `reading-summary-fiction.md` |
| 技术/工具书 | `technical` | 编程、架构、工程实践 | `reading-summary-technical.md` |
| 通用/非虚构 | `general` | 商业、社科、心理、历史、人文 | `reading-summary-general.md` |

### 类型识别规则

**优先级：用户显式 > 关键词推断 > 主动询问**

1. **用户显式指定**：如"开始读技术书《xxx》"、"开始读小说《xxx》"
2. **关键词推断**（仅作为提示，必要时仍需确认）：
   - fiction：小说、长篇、短篇、推理、科幻、奇幻、剧本、散文
   - technical：编程、架构、源码、框架、算法、系统设计、某技术名（如 Kubernetes、Rust）
   - general：商业、管理、心理、经济、历史、传记（非虚构）、社科
3. **不确定时主动询问**：给出三个选项，让用户选

## 书名主键规则（Sanitize）

用于构建文件名 `reading-{key}.md`，避免特殊字符导致的文件系统问题。

**处理步骤**（按顺序）：
1. 去除书名号：`《》「」『』""''`
2. 去除文件系统非法字符：`/ \ : * ? " < > |`
3. 将中英文冒号、斜杠替换为连字符：`： ：/` → `-`
4. 去除首尾空格
5. 将内部连续空格替换为单个连字符 `-`
6. 保留：中英文字符、数字、连字符、下划线、点

**示例**：

| 原始书名 | Sanitized Key | 文件名 |
|----------|---------------|--------|
| 《三体》 | `三体` | `reading-三体.md` |
| 《三体 II：黑暗森林》 | `三体-II-黑暗森林` | `reading-三体-II-黑暗森林.md` |
| 《The Pragmatic Programmer》 | `The-Pragmatic-Programmer` | `reading-The-Pragmatic-Programmer.md` |
| 《代码大全 2》 | `代码大全-2` | `reading-代码大全-2.md` |

## 核心流程

### 流程一：开始阅读（触发："开始读 xxx"）

**分两段：先启动问询，再创建文件。**

#### 第一段：启动问询

```
用户说："开始读《xxx》"
    ↓
1. 解析书名（提取《》内内容或纯文本）
2. 识别书籍类型（显式 > 关键词推断 > 询问，见"类型识别规则"）
3. AI 先主动给出"为什么想读这本书"的 3-5 种可能性
   - 基于书名、已知的作者/主题
   - 用"可能是因为..."的口吻，让用户挑选或补充
4. 一次性列出启动问询清单（共 7 项），让用户统一回复：
   ① 阅读工具（Kindle / 微信读书 / 纸质书 / ...）
   ② 阅读场景（通勤 / 睡前 / ...）
   ③ 每次阅读量（时间或章节/页数）
   ④ 每周阅读几次
   ⑤ 预估阅读周期（几周）
   ⑥ 为什么想读这本书（从第 3 步选项里选 / 或自己说）
   ⑦ 读之前有哪些想法 / 期待
5. 用户回答不全时，只记录已回答项，缺项留空（禁止编造）
```

#### 第二段：创建文件与登记

```
启动问询完成后：
    ↓
1. 按 Sanitize 规则生成主键
2. 创建专项文件 `memory/reading-{key}.md`
   - 读取 references/reading-template.md
   - 填充：书名、作者、类型、开始日期（今天）
   - 填充"🚀 阅读计划"区块（7 项启动问询答案）
3. 在当日 `memory/YYYY-MM-DD.md` 中添加任务项
   - 格式：`- [ ] 阅读《{书名}》`
4. 回复确认："已创建《xxx》读书记录（类型：xxx，预计 N 周读完），随时记录进度～"
```

**执行要点：**
- **启动问询不能跳过**：即使用户只说"开始读《xxx》"，也要先走启动问询再建文件
- **动机推测要具体**：不要给"可能对这个主题感兴趣"这种泛泛话，要结合书名/作者给出具体猜测
- **问询格式**：清单式一次列完，不要一个个追问，尊重用户时间
- **禁止编造**：用户没答的项留空，不要替他填

---

### 流程二：日常记录（触发："读了 xxx"/"完成 xxx 阅读"）

```
用户说："读了《xxx》第 X 章" 或 "完成《xxx》阅读"
    ↓
1. 解析书名和进度信息
2. 按 Sanitize 规则定位专项文件 `memory/reading-{key}.md`
3. 更新进度表（新增一行：日期、章节、进度、备注）
4. 更新章节笔记（如用户提供）
5. 更新感想（如用户提供）
6. 更新当日 `memory/YYYY-MM-DD.md`：找到任务项打勾 + 补充进度
7. 判断是否读完（关键词："读完"/"完成"/"全书完"）
   - 如读完 → 主动问："要不要现在做个总结？"
8. 回复确认："已记录《xxx》进度"
```

**执行要点：**
- 双轨更新：同时更新专项文件和当日记录
- 读完触发：主动询问是否总结，但不强制

---

### 流程三：读后总结（触发："总结 xxx"/"聊聊 xxx"）

```
用户说："总结《xxx》"
    ↓
1. 按 Sanitize 规则定位 `memory/reading-{key}.md`
2. 读取专项文件，获取类型字段
3. 读取 memory/ 下相关日期的记录（补充感想）
4. 根据类型选择对应 summary 模板：
   - fiction → reading-summary-fiction.md
   - technical → reading-summary-technical.md
   - general → reading-summary-general.md
5. 生成结构化总结（Markdown）
6. 获取飞书 folder_token（见下方规则）
7. 创建飞书文档
8. 回复链接 + "总结完成，任务可以打勾了～"
```

#### 飞书 folder_token 获取规则

```
1. 从 agent memory 中查找读书总结的 folder_token（reference 类型记忆）
   - 找到 → 直接使用
   - 找不到 → 跳到步骤 2
2. 询问用户："飞书文档创建到哪个文件夹？"
   - 用户给 folder_token → 使用，并询问是否保存为默认
   - 用户说"随便"/"我的空间" → 创建到根目录（folder_token 留空）
3. 用户同意保存 → 作为 reference 类型记忆存入 agent memory
```

---

### 流程四：模板查看（触发："阅读模板"/"看总结模板"）

```
用户说："看阅读模板" / "看小说总结模板" / "看技术总结模板"
    ↓
1. 识别用户想看的模板类型
2. 读取并展示对应文件：
   - 进度模板 → references/reading-template.md
   - 小说总结 → references/reading-summary-fiction.md
   - 技术总结 → references/reading-summary-technical.md
   - 通用总结 → references/reading-summary-general.md
3. 如未指定类型，列出可选项让用户选
```

## 文件结构

```
skills/reading-workflow/
├── SKILL.md
└── references/
    ├── reading-template.md              # 进度追踪模板（通用）
    ├── reading-summary-fiction.md       # 小说总结模板
    ├── reading-summary-technical.md     # 技术书总结模板
    └── reading-summary-general.md       # 通用/非虚构总结模板

memory/
├── reading-{key}.md                     # 每本书的专项记录
└── YYYY-MM-DD.md                        # 每日任务清单
```

> 飞书 folder_token 等偏好配置存入 agent 的持久 memory（reference 类型），不放在项目 memory/ 目录。

## 注意事项

1. **禁止编造**：只记录用户亲口说的内容，不推断、不扩写
2. **启动问询不可跳**：流程一必须先问清 7 项阅读计划再建文件
3. **双轨记录**：同时更新专项文件和当日记录，防会话重启丢失
4. **类型先定**：开始读时就确定类型，避免总结时类型不清
5. **Sanitize 一致**：开始/记录/总结都用同一 sanitize 规则定位文件
6. **读完触发**：主动询问是否总结，但由用户决定
7. **folder_token**：先查 agent memory，没有再问用户；用户同意后存为 reference 记忆

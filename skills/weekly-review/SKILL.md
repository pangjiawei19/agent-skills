# weekly-review - 周工作总结 Skill

## 📋 功能描述

基于每日任务记录，生成周工作总结飞书文档，包含工作任务、学习与成长、生活内容、关键词提炼、复盘反思等模块。

---

## 🎯 触发条件

1. 用户说"做周总结"、"本周总结"、"上周总结"、"周工作总结"等
2. 每周日 18:00（Cron 自动触发）
3. 用户要求整理本周/某周工作、学习、生活内容

---

## 🗓️ 周期定义

- **"本周"/"周总结"** = 本周一 00:00 → 本周日 23:59
- **"上周"/"上周总结"** = 上周一 → 上周日
- **"上上周"** = 依此类推
- 周末跨天的任务按实际记录所在日期归属

手动触发时如果语义不明（如周三说"做个总结"），**先向用户确认周期**再执行。

---

## 📁 文件结构

```
skills/weekly-review/

└── template.md        # 周总结文档模板（唯一事实源）
```

文档模板完整内容见 `template.md`，本文件不再重复嵌入，避免漂移。

## 🚦 前置检查

执行前必须检查：

1. **folder_token 是否由用户提供**
   - 首次执行或缺失时，向用户索要"个人规划"目录的 `folder_token`
   - 未提供时**直接返回失败提示**，不要猜测、不要硬编码：
     > 缺少飞书目录 folder_token，请提供后再执行。可通过 `feishu_search_doc_wiki` 搜索"个人规划"获取。
   - 建议用户将 token 存在本地配置文件（非 git 跟踪），每次会话传入

2. **周期是否明确**
   - 手动触发且语义含糊时，先确认周期

3. **memory 文件是否齐全**
   - 某天缺失时直接在文档标注"无记录"，不要编造

## 🔄 执行流程

### 1. 收集每日记录

按周期读取 `memory/YYYY-MM-DD.md`。

### 2. 分类整理

| 类别 | 识别规则 |
|------|---------|
| 💻 **工作任务** | 项目开发、代码、会议、文档、自动化任务等 |
| 📚 **学习与成长** | 读书、技术调研、英语学习、课程等 |
| 🏠 **生活内容** | **仅记录用户主动分享的内容**（运动、做饭、家庭等） |

**边界示例：**
- ✅ 读《白夜行》→ 学习
- ✅ 技术文档阅读 → 学习
- ✅ 用户说"今天去跑步了 5km" → 生活（主动分享）
- ❌ 用户随口提"中午吃了面" → 不算生活内容（非主动分享的记录）
- ❌ 用户说"累了" → 不算生活内容（情绪表达非事项）
- ❌ 从代码/工作推断"应该加班了" → **禁止推断**

### 3. ⚠️ 禁止编造原则（核心红线）

- **只基于 memory 中实际记录的内容**，不要为了让表格好看而虚构任务
- 某天无记录 → 标注"无记录"
- 内容不足 → 直接写少，不要扩写
- 不确定的归类 → 保留原始描述，不强行分类

### 4. 提炼关键词

3-5 个关键词，每个带简短说明（为什么重要）。

**示例：**
1. **MCP 架构迁移** - 完成 EKS MCP 替换 Tools，生产验证通过
2. **Family-Assistant 收尾** - 文档、代码规范整理完毕，项目告一段落
3. **自动化稳定运行** - GitHub Trending、InfoQ 资讯每日推送无故障

### 5. 选取复盘事项（1-3 件）

**选取标准：**
- ✅ 有重大进展/突破
- ✅ 有计划与实际的差异
- ✅ 有重要经验教训
- ❌ 常规执行、无特殊事项的不复盘

**真实复盘示例（good example）：**

> **事项：EKS MCP 迁移生产上线延后 2 天**
>
> - **事实：** 计划周三上线，实际周五上线。原定周三完成替换、周四观察、周五收尾，实际因 Tools 调用兼容性问题重写了 3 个接口。
> - **原因/分析：** 预研阶段只验证了 happy path，没测边界情况（空参数、超时重试），低估了 Tools → MCP 协议差异的改造量。
> - **改进措施：** 后续协议迁移类任务，预研阶段必须覆盖 3 类异常场景；迁移前先写一份兼容性 checklist 再估工时。

**反例（不要这样写）：**

> **事项：本周完成了很多工作**
> - **事实：** 做了 A、B、C
> - **分析：** 效率很高
> - **改进：** 继续保持

（这种是流水账，没有真正反思，不如不写）

### 6. AI 视角观察（可选）

只观察协作模式、工作习惯，不评判个人选择。没有新认知就写"暂无新观察"。

### 7. 创建飞书文档

```yaml
title: "周工作总结 (YYYY-MM-DD ~ YYYY-MM-DD)"
markdown: <按 template.md 填充的完整内容>
folder_token: <用户提供>
```

### 8. 发送飞书通知

```json
{
  "msg_type": "text",
  "content": "{\"text\": \"老大，本周总结已生成：[文档链接]\"}"
}
```

## ⚠️ 注意事项

1. **周期定义：** 周一 → 周日（见"周期定义"章节）
2. **生活内容边界：** 仅记录用户主动分享，不主动追问、不推断
3. **复盘粒度：** 1-3 件重要事情，不事事复盘
4. **folder_token：** 由用户提供，未提供直接失败
5. **禁止编造：** 只基于 memory 实际记录
6. **执行时间：** 周日 18:00（北京时间）自动触发

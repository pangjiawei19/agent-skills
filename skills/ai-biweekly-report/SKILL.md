# AI 工程半月报

## 执行流程

### Step 1: 确认参数

从用户请求中提取以下参数：

- **时间范围**：上半月（1-15 日）或下半月（16-月底），或自定义范围
- **飞书 folder_token**：用于创建飞书文档的目标文件夹 token

如果用户未提供 `folder_token`，**必须主动询问**用户提供。如果用户明确表示没有或拒绝提供，**返回失败提示**：

> 无法创建飞书文档：需要提供飞书文件夹的 `folder_token` 才能创建文档。请提供目标文件夹的 token 后重试。

**不要猜测或使用任何默认值。**

### Step 2: 多数据源搜索（扩大候选池）

搜索 8 大数据源，预期 50-80 条候选。**搜索关键词中的日期必须动态匹配当前报告的时间范围。**

数据源分为核心和备选两级：

**核心数据源**（必须搜索）：

| 数据源 | 搜索关键词示例 | 预期产出 |
|--------|---------------|----------|
| Arxiv | `cs.CL cs.LG cs.AI {month} {year}` | 10-15 条 |
| Hacker News | `AI LLM {month} {year}` | 10-15 条 |
| 厂商博客 | `OpenAI Anthropic Google AI {month} {year}` | 5-8 条 |
| GitHub Trending | `AI machine learning trending` | 5-10 条 |

**备选数据源**（核心数据源产出不足时补充）：

| 数据源 | 搜索关键词示例 | 预期产出 |
|--------|---------------|----------|
| Hugging Face | `new model release {month} {year}` | 5-8 条 |
| TechCrunch | `AI startup funding {month} {year}` | 5-10 条 |
| 向量数据库 | `Pinecone Weaviate Milvus {month} {year}` | 3-5 条 |
| AI 工程实践 | `RAG production deployment {year}` | 5-8 条 |

**降级策略**：
- 如果某个核心数据源搜索失败或返回 0 结果，跳过并记录，从备选数据源补充
- 如果所有核心数据源均失败，告知用户网络或搜索服务可能有问题

**工具**：`web_search`（每次 count=10，分批次搜索）

### Step 3: 日期过滤与核实

采用**两阶段过滤**，减少不必要的 web_fetch 调用：

**阶段 A：基于搜索摘要的初步过滤**
- 从搜索结果的标题、摘要、URL 中提取日期信息
- 明显超出时间范围的直接排除（如 URL 含 `/2024/`、标题含去年月份等）
- 无法判断日期的保留进入阶段 B

**阶段 B：web_fetch 精确核实**
- 仅对阶段 A 中**日期不明确**的候选条目执行 web_fetch
- 从原文中提取发布日期，过滤不在时间范围内的内容
- 记录原文链接

**时间范围**：
- 上半月报：当月 1-15 日
- 下半月报：当月 16-月底

### Step 4: 按优先级筛选 Top 10

筛选标准（弹性比例）：

| 类别 | 目标范围 | 说明 |
|------|---------|------|
| 产业界 | 4-6 条 | 新模型 API、框架更新、向量数据库、云厂商服务 |
| 学术界 | 2-4 条 | 有开源代码、新架构、提升 RAG/微调效果 |
| 高价值实践 | 1-3 条 | GitHub 热门项目、工程化最佳实践 |

总数控制在 8-12 条。**宁缺毋滥**，高质量资讯不足时不凑数，告知用户："本期高质量资讯较少，仅筛选出 X 条"。

### Step 5: 深度分析（逐条处理）

每条资讯按以下格式输出：

```markdown
### [序号] 标题

**来源**：数据源 | 发布日期
**原文链接**：URL

- **核心情报 (What)**：一句话概括发生了什么

- **技术原理解析 (How)**：
  - 技术逻辑/实现机制
  - 关键创新点
  - 性能指标

- **应用场景 (Where)**：
  - 可以做什么产品
  - 解决什么痛点

- **工程化价值 (Why)**：
  - 对开发者意味着什么
  - 成本/延迟/架构影响
  - 权衡和注意事项
```

### Step 6: 转型指导与行动建议

包含三个部分：

#### 技术栈迁移建议
| 项目 | 推荐理由 | 优先级 | 链接 |
|------|----------|--------|------|
| 项目名 | 为什么值得 Clone | 高/中/低 | GitHub 链接 |

#### 架构思考
- 数据库选型影响
- 微服务拆分建议
- API 网关/缓存层设计

#### 学习重点
- 下半月建议补充的知识
- 推荐学习资源

### Step 7: 生成飞书文档

1. 用 `feishu_create_doc` 在用户提供的 `folder_token` 下创建完整文档
2. 用 `message` 发送摘要消息（含文档链接）

输出格式参见 [references/template.md](./references/template.md)。

## 错误处理

1. **候选资讯不足**：有几条写几条，不凑数，告知用户
2. **web_fetch 失败**：尝试备用 URL 或跳过该条，记录失败原因
3. **日期无法核实**：保守处理，归入"日期不明"类别，在文档中标注
4. **飞书创建失败**：提示用户检查授权和 folder_token，提供 Markdown 版本作为备选

## 注意事项

1. **时间范围严格核实**：每条资讯必须确认在指定时间范围内
2. **原文链接必附**：方便用户快速查看原文
3. **工程视角**：始终从后端工程师转型 AI Engineer 的视角分析
4. **避免重复**：半月报周期确保不重复报道相同内容

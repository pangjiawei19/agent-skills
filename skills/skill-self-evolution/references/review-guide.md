# 阶段复盘指南

用于读取某个目标 Skill 的运行轨迹，判断是否需要修改 Skill。

## １．收集证据

读取目标 Skill 对应目录：

- 项目 Skill：`skill-self-evolution/skill-runs/<skill-name>/`
- 全局 Skill：`~/.agents/skill-self-evolution/skill-runs/<skill-name>/`

优先查看最近尚未复盘的运行轨迹。不要只看单条记录就下结论。

## ２．分类问题

把观察分成四类：

- 偶发失误：单次 LLM 判断、表达或操作失误，暂不改 Skill。
- 流程缺口：步骤顺序、停顿点、确认点、路由缺失。
- 模板缺口：输出字段、命名、记录格式、检查项缺失。
- 高风险问题：误删、覆盖、污染数据、错误发布、错误发消息、错误修改 Skill。

## ３．触发判断

满足任一条件，生成阶段复盘：

- 累计 ５ 次新的运行轨迹；
- 同一类问题出现 ２ 次；
- 出现高风险问题；
- 用户明确要求复盘目标 Skill。

否则只记录观察，不建议修改。

## ４．提出候选改动

每条候选改动必须写清：

- 问题是什么；
- 证据来自哪些运行轨迹；
- 建议改哪里；
- 预期收益是什么；
- 潜在风险是什么；
- 是否建议本轮执行。

候选改动不是执行命令。等待用户确认后再修改。

## ５．不采纳项

对暂不修改的观察，也要写清原因：

- 证据只有一次；
- 影响很小；
- 更像执行失误而不是规则缺口；
- 修改可能导致目标 Skill 过拟合；
- 可以先通过下次提醒观察。

## ６．修改后回归

如果用户确认修改目标 Skill，改完后至少验证：

- 目标 Skill 的触发描述没有变窄或变歪；
- 主流程仍能跑通；
- 原有产物格式没有被破坏；
- 高风险操作仍有确认或保护；
- 运行轨迹仍会写到正确目录；
- 候选改动不会绕过用户确认直接落地。

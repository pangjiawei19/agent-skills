# Skill Self Evolution

这是一个用于**改造其他 Skill** 的 Skill。它会修改目标 Skill，让目标 Skill 自己拥有一套轻量、可控、可复盘的自进化能力。

加装完成后，`skill-self-evolution` 本身不参与目标 Skill 的日常执行。目标 Skill 会独立运行，并在自己的规则里完成运行轨迹记录、用户反馈采集、阶段复盘、候选改动和回归检查。

## 怎么用

### １．给目标 Skill 加装自进化能力

```text
请用 skill-self-evolution 给 weekly-review 加装自进化闭环。
```

执行时会先读取目标 Skill，识别它的主流程、产物和高风险动作，然后提出加装方案。用户确认后，才会修改目标 Skill。

### ２．加装完成后使用目标 Skill

之后继续正常使用目标 Skill，不需要每次再调用 `skill-self-evolution`。

例如加装对象是 `weekly-review`，后续仍然直接使用：

```text
请用 weekly-review 生成本周复盘。
```

这时 `weekly-review` 会按已加装的规则，在运行结束后记录轨迹、轻问用户反馈，并在满足条件时支持阶段复盘。

### ３．复盘目标 Skill 的运行轨迹

```text
请复盘 weekly-review 最近的运行轨迹，看看有没有值得改 Skill 的地方。
```

复盘会输出候选改动和不采纳项。是否真的修改目标 Skill，仍然需要用户确认。

## 它改造后的目标 Skill 会获得什么

- 每次运行后记录结构化运行轨迹。
- 运行结束写轨迹前，轻量询问用户使用感受或改进想法。
- 累积多次轨迹后做阶段复盘。
- 将问题、用户反馈和改进想法沉淀为候选改动。
- 在用户确认后小步修改自身规则、模板或保护边界。
- 修改后执行回归检查，避免自进化破坏原有能力。

目标 Skill 获得的是“基于证据持续改进”的能力，不是“自动随意重写自己”的能力。

## 适用场景

- 想给某个 Skill 增加长期迭代能力。
- 想记录某个 Skill 每次运行后的质量、问题和用户反馈。
- 想基于多次运行轨迹判断是否应该修改 Skill。
- 想把用户的改进想法沉淀为候选改动，但不希望 AI 直接自动改规则。
- 想给高风险 Skill 增加复盘、确认和回归检查机制。

## 不适用场景

- 只想马上完成一次业务任务。
- 只根据一次偶发执行失误就修改 Skill。
- 希望 Skill 不经用户确认自动重写自身规则。
- 希望完整保存对话转录。运行轨迹只记录结构化摘要。

## 工作闭环

加装后的目标 Skill 按这个闭环独立运行：

```text
运行轨迹 → 用户反馈 → 阶段复盘 → 候选改动 → 用户确认 → 小步修改 → 回归检查
```

### 运行轨迹

目标 Skill 每次使用结束后，写入一份结构化运行轨迹，通常包含：

- 用户意图与目标；
- 执行路径；
- 质量检查；
- 用户反馈；
- 问题与观察；
- 候选改动；
- 下次注意。

写轨迹前会轻量询问用户一次使用感受或改进想法。用户不回答则跳过，不反复追问。

### 阶段复盘

满足以下任一条件时，生成阶段复盘：

- 累计 ５ 次新的运行轨迹；
- 同一类问题出现 ２ 次；
- 出现高风险问题；
- 用户明确要求复盘目标 Skill。

阶段复盘只归纳证据和候选改动，不直接执行修改。

### 小步修改和回归检查

只有用户确认后，才修改目标 Skill。单次修改最多落地 １～３ 条规则，并优先修改局部流程、模板或保护规则，避免重写整个 `SKILL.md`。

修改后按回归检查清单验证触发、路由、输入、主流程、输出、风险、反馈、记录和复盘边界。

```text
skills/skill-self-evolution/
├── SKILL.md                         # Skill 主说明，agent 读取
├── README.md                        # 本文件，给人看的说明
├── agents/
│   └── openai.yaml                  # OpenAI / Codex 展示信息
└── references/
    ├── install-guide.md             # 给目标 Skill 加装自进化机制的步骤
    ├── review-guide.md              # 从运行轨迹做阶段复盘的方法
    └── templates.md                 # 运行轨迹、阶段复盘、回归检查模板
```

## 数据目录

项目 Skill 默认写入项目内：

```text
<project-root>/skill-self-evolution/skill-runs/<skill-name>/
<project-root>/skill-self-evolution/skill-reviews/<skill-name>/
```

全局 Skill 默认写入：

```text
~/.agents/skill-self-evolution/skill-runs/<skill-name>/
~/.agents/skill-self-evolution/skill-reviews/<skill-name>/
```

用户可以指定其他路径，但需要在加装方案中明确说明。

## 补充用法

### 手动记录一次运行轨迹

```text
这次 weekly-review 用完了，帮我按自进化模板记录运行轨迹。
```

记录时只保存结构化摘要，不保存完整对话。

## 安全边界

- 不根据单次运行直接修改目标 Skill。
- 候选改动默认只记录，不自动执行。
- 用户反馈只进入运行轨迹和候选改动，不绕过阶段复盘和用户确认。
- 同类问题出现 ２ 次以上，或出现高风险问题，才建议进入复盘。
- 单次复盘最多让 １～３ 条建议进入本轮修改。
- 修改前说明证据、改动位置、预期收益和潜在风险。
- 修改后执行回归检查。

## 版本管理

目标 Skill 可以在 `SKILL.md` frontmatter 顶层维护 `version` 字段。只有用户明确要求启用版本管理，或目标 Skill 已经存在 `version` 字段时，才维护该字段。

默认采用语义化版本：

- 破坏性或大范围流程调整升主版本；
- 新增能力升次版本；
- 文案、模板细节、局部保护规则升补丁版本。

## 维护建议

- 优先让运行轨迹积累证据，不急着改规则。
- 复盘时区分偶发失误、流程缺口、模板缺口和高风险问题。
- 不要把一次用户偏好过度泛化为全局规则。
- 改动越靠近高风险动作，越需要明确确认点和回归检查。

# 日常工作流程

## 核心流程

```
🌅 早上老大说"开工"
    ↓
我主动问老大：今天要做什么？有几件重点事？
（不从 git / 代码 / 昨日记录推断，一律问老大）
    ↓
☀️ 白天：老大主动说"完成了 xxx"
    ↓
我简单记录，不追问细节（降低干扰）
    ↓
🌙 老大说"收工"
    ↓
我按「工作 / 学习 / 生活」分类拉清单（每类内按完成时间升序）
→ 老大确认/补充 + 告诉我今天有没有其他重要事
    ↓
我推荐 1-3 件值得复盘的事 → 老大确认或指定其他
    ↓
按模板复盘：事实 → 原因 → 改进措施
    ↓
把复盘结果完整发给老大 → 老大确认/修改/补充
    ↓
沉淀到 memory/YYYY-MM-DD.md（未确认前禁止写文件）
```

## ⚠️ 禁止编造原则（核心红线）

- **只记录老大亲口说的内容**，不推断、不扩写、不脑补
- 开工拉清单 → 问老大，禁止从 git log / 代码改动 / 昨日未完成项自动推断今日任务
- 白天记录 → 老大说什么记什么，不为了"看起来饱满"补充细节
- 收工复盘 → 某项没信息就写少，不要用"应该、可能、大概"填空
- 与 `weekly-review` 保持一致红线：内容不足直接写少，不编造

## 任务记录规则

| 环节 | 执行方式 |
|------|----------|
| **开工清单** | 我主动问老大今天做什么，不自行推断 |
| **任务记录** | 老大主动告诉我，我简单记（不追问） |
| **完成时间** | 记录"完成了 xxx"时同步记下当前时间（HH:MM，24 小时制），时间以收到消息时刻为准 |
| **任务分类** | 每条任务标注类别：`[工作]` / `[学习]` / `[生活]`，分类规则与 weekly-review 一致 |
| **追问补细节** | 晚上"收工"后一起补 |
| **复盘触发** | 我按重要性推荐 1-3 件 + 老大可指定 |
| **复盘确认** | 复盘草稿必须先完整发给老大，等老大确认/修改后才能写入 memory 文件 |
| **实时写文件** | 每次说"完成了 xxx"，立即更新 memory 文件（防会话重启丢失）；复盘部分例外，须等确认 |

### 任务条目格式

每条完成任务统一按此格式写入 memory 文件：

```
- [类别] HH:MM {任务描述}
```

**示例：**

```
- [工作] 10:23 完成 EKS MCP 接口替换并通过联调
- [学习] 14:05 读完《白夜行》第 3 章
- [生活] 19:40 下班后去跑了 5km
```

分类不确定时保留原始描述 + 问老大，禁止硬塞。

## 收工拉清单格式

收工时按 **工作 / 学习 / 生活** 三类展示，每类内部按 `HH:MM` 升序排列。某类无内容直接写"无"，不补内容。

```markdown
### 💻 工作
- 10:23 完成 EKS MCP 接口替换并通过联调
- 16:50 搞定 Family-Assistant 文档收尾

### 📚 学习
- 14:05 读完《白夜行》第 3 章

### 🏠 生活
- 19:40 下班后去跑了 5km
```

展示完清单后再进入"老大确认/补充 → 推荐复盘事项"环节。

## 复盘模板

```markdown
### 事项 N：{事项名称}

**事实：**
- 计划了什么
- 实际完成了什么
- 关键时间节点

**原因/分析：**
- 为什么会有差异
- 原因是什么（客观/主观）
- 有什么意外发现

**改进措施：**
- 下次怎么做
- 需要沉淀到什么文件/流程
```

## 复盘确认环节

复盘不是 AI 一锤定音的事 —— 事实、原因、改进措施都可能有我没观察到或理解偏差的地方，所以草稿必须先过老大这一关，避免把错的东西沉淀进 memory 后以讹传讹。

流程：

1. 我按模板写出完整的复盘草稿（所有事项一次性给出，不分批）
2. 以 markdown 形式直接贴在对话里发给老大，结尾加一句类似：**"以上复盘草稿请老大过目，有需要修改/补充/删减的地方直接说，确认无误后我再写入今日 memory 文件。"**
3. 等老大明确回复（"OK / 可以 / 写入吧" 或带修改意见）
4. 如果有修改意见 → 按意见改完再发一版 → 继续等确认（可多轮）
5. 收到确认后，才把最终版本写入 `memory/YYYY-MM-DD.md`

**禁止：**
- 草稿还没发就直接写文件
- 老大还没回复就默认"应该没问题"继续沉淀
- 写入文件时偷偷加草稿里没有的内容

# 日常工作流程

## 核心原则：一天一份清单，只改不重建

整天只维护**一份** 任务清单：

- 开工拉清单 = 初始化这份文件（按工作/学习/生活三类写入待办）
- 白天"完成了 xxx" = 在**同一份清单**里打勾、加时间、补细节
- 收工 = 直接拿这份文件做展示和复盘依据

之所以强调"只改不另起"，是因为之前实战中出现过：AI 会新起一段"已完成任务"来记录，导致同一件事在文件里出现两次、状态对不上，一天下来信息非常混乱。整天动的都是同一份清单，信息源才唯一、可信。

## 核心流程

```
🌅 老大说"开工"
    ↓
我问老大：今天工作 / 学习 / 生活分别要做什么？
（不从 git / 代码 / 昨日记录推断）
    ↓
老大报清单 → 我立刻写入 memory/YYYY-MM-DD.md
（按三类分区，全部 - [ ] 未完成态；老大说出口即视为确认，无需再复述）
    ↓
☀️ 白天：老大说"完成了 xxx"
    ↓
我在同一份文件里找对应条目：- [ ] → - [x]，在描述末尾追加 `(HH:MM)`，原描述保持不动
老大顺带讲的细节 → 作为子 bullet 追加在该条目下
若该任务不在早上清单里 → 直接追加到对应大类末尾（已是 [x] 完成态）
    ↓
🌙 老大说"收工"
    ↓
我把当前清单原样贴出（已完成 + 未完成一起，按三类分）
→ 老大确认 / 补充
    ↓
我按重要性推荐 1-3 件值得复盘的事 → 老大确认或指定其他
    ↓
按模板写复盘草稿 → 发给老大确认
    ↓
确认后才把复盘段落追加到 memory/YYYY-MM-DD.md
```

## 文件结构

`memory/YYYY-MM-DD.md` 整天都长这样（动态演进）：

```markdown
# 2026-04-23

## 📋 今日任务清单

### 💻 工作
- [ ] EKS MCP 接口替换
- [x] Family-Assistant 文档收尾 (10:23)
  - 评审通过，已归档

### 📚 学习
- [ ] 读《白夜行》第 3 章

### 🏠 生活
- [x] 跑 5km (19:40)

## 📝 今日复盘（收工确认后才写入）
...
```

## 各环节执行规则

| 环节 | 执行方式 |
|------|----------|
| **开工初始化** | 主动问老大三类任务，老大说什么就写什么，**立刻落盘**（无需让老大复述确认） |
| **白天完成** | 找到原条目打勾 + 在描述末尾追加 `(HH:MM)` + 可追加子 bullet 补细节，**绝不新开段落** |
| **计划外任务** | 直接在对应大类末尾追加为 `- [x] 描述 (HH:MM)`，不做"临时"标记——今天完成的就是今天完成的 |
| **完成时间** | 以收到"完成了 xxx"消息的时刻为准，HH:MM，24 小时制，放在描述末尾用小括号括起来 |
| **条目匹配** | 老大的描述和清单项字面不一致时，由我自行判断是否为同一项（合理匹配即可，不找老大确认）；若明显对不上任何清单项，按"计划外"追加 |
| **任务分类** | 工作 / 学习 / 生活，与 `weekly-review` 一致 |
| **追问细节** | 收工后一起补，白天不打扰 |
| **复盘触发** | 我按重要性推荐 1-3 件 + 老大可指定 |
| **复盘写入** | 草稿必须先发给老大，确认后才能追加到文件 |

## 条目格式

```
未完成：- [ ] {描述}
已完成：- [x] {描述} (HH:MM)
补细节：- [x] {描述} (HH:MM)
             - 追加的细节说明
```

**设计意图**：描述放前面最突出（一眼看到在做什么），时间作为附属信息放小括号里；未完成的条目自然就没有括号，状态一目了然。

真实示例：

```markdown
### 💻 工作
- [ ] 调研 B 方案可行性
- [x] EKS MCP 接口替换 (10:23)
  - 联调通过，延迟 200ms → 80ms
- [x] Family-Assistant 文档收尾 (16:50)

### 📚 学习
- [x] 读完《白夜行》第 3 章 (14:05)

### 🏠 生活
- [x] 跑 5km (19:40)
```

## 收工拉清单

直接把当前 memory 文件中三类内容**原样贴出**（包括未完成的 `- [ ]`），不改结构、不单独开"未完成"小节。未完成就是未完成，呈现即可。

```markdown
### 💻 工作
- [ ] 调研 B 方案可行性
- [x] EKS MCP 接口替换 (10:23)
- [x] Family-Assistant 文档收尾 (16:50)

### 📚 学习
- [x] 读完《白夜行》第 3 章 (14:05)

### 🏠 生活
- [x] 跑 5km (19:40)
```

某类整天都无内容就写"无"。展示完进入老大确认 / 补充 → 复盘推荐环节。

## 复盘模板

```markdown
### 事项 N：{事项名称}

**事实：**
- 计划了什么
- 实际完成了什么
- 关键时间节点

**原因 / 分析：**
- 为什么会有差异
- 原因是什么（客观 / 主观）
- 有什么意外发现

**改进措施：**
- 下次怎么做
- 需要沉淀到什么文件 / 流程
```

## 复盘确认环节

复盘不是 AI 一锤定音的事 —— 事实、原因、改进措施都可能有我没观察到或理解偏差的地方，所以草稿必须先过老大这一关，避免把错的东西沉淀进 memory 后以讹传讹。

流程：

1. 按模板写完整草稿（所有事项一次性给出，不分批）
2. 以 markdown 形式贴在对话里，结尾加一句："**以上复盘草稿请老大过目，有需要修改 / 补充 / 删减的地方直接说，确认无误后我再写入今日 memory 文件。**"
3. 等老大明确回复（"OK / 可以 / 写入吧"或带修改意见）
4. 有修改意见 → 改完再发一版 → 继续等确认（可多轮）
5. 确认后才把最终版本追加到 `memory/YYYY-MM-DD.md` 的复盘段落

## ⛔ 禁止自行发挥（红线，必看）

这份 skill 存在的意义就是让记录**稳定、可信、不混乱**。下面每条都有过实战翻车的来由，不是装饰：

1. **禁止自己起新清单 / 新段落**
   一天只有一份 `任务清单`。完成任务永远是更新原条目，绝不因为"新起一段看起来更整齐"、"汇总一下更好看"而另开区域。这是本次改版的根本起因。

2. **禁止编造任务项**
   开工清单只记录老大亲口说的。禁止从 git log、代码改动、昨日未完成项、日历事件等推断"老大今天应该也会做 X"并偷偷写进清单。

3. **禁止偷改 / 覆盖原描述**
   完成时只把 `[ ]` 改成 `[x]`、在描述末尾追加 `(HH:MM)`，**原描述一个字都别动**。补充细节一律用子 bullet 追加，不要改写原文案——老大需要看到自己早上定的是什么样。

4. **禁止脑补细节**
   老大说"完成了 EKS 接口替换"就只写这一句。不许扩写成"完成了 EKS MCP 接口替换，性能应该有提升"。没说的一个字都不要加。

5. **禁止拿昨日清单冒充今日**
   每天开工都要重新问老大。不要因为昨天写了什么、或者有未完成项，就默认今天也做同样的事。

6. **禁止在复盘草稿未确认前写入文件**
   任务条目可实时写入（因为是老大原话），复盘是 AI 的二次加工，必须先让老大审过。

7. **内容不足时就写少**
   某项老大没讲清楚，就记已有部分，禁止用"应该 / 可能 / 大概"填空。与 `weekly-review` 红线一致。

一句话总结：**老大说的 → 照抄照记；老大没说的 → 一个字都不加；清单只有一份 → 一直改它，不另起。**

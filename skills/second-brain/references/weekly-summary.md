# 周总结 — 编辑模式 + 教练衔接

## 语义

用户说"周总结" / "周报" / "本周总结" / "上周总结"= 把一周的 daily 记录汇总成一份周报草稿；草稿用户确认后写入 `reports/`，末尾询问是否进入 `+reflect` 深度反思。

## 模式定位（CRITICAL）

周总结使用**编辑模式**，介于 daily 秘书模式和 brain 教练模式之间：

- ✅ 可以：合并同类项、概括表述、归纳关键词、给出克制的整体观察
- ❌ 不能：新增 daily 里没出现过的事实、推断未记录的细节、写空话套话

红线对应：
- 红线 2（禁编造）、红线 4（禁脑补）、红线 6（未确认禁写入）、红线 7（内容不足就写少）**完全保留**
- 红线 3（禁偷改原描述）**放宽**：允许合并/概括表述（如把一周 5 条"写 X 模块"合并为"完成 X 模块开发"），但不得歪曲原意

## 启动流程

1. **加载 profile**（用 Read 工具，顺序如下；任一文件不存在时跳过，不报错）：
   - `~/second-brain/profile/user.md`
   - `~/second-brain/profile/soul.md`
   - `~/second-brain/profile/patterns.md`
   - `~/second-brain/profile/memory.md`

2. **周期确认**：判定用户指的是哪一周。
   - 用户明确说"上周总结" / "上周报" → 周期 = 上周一 00:00 → 上周日 23:59，跳过询问
   - 用户明确说"本周总结" / "本周报" → 周期 = 本周一 00:00 → 今天 23:59（截至执行时刻；如今天是周一，提醒用户"今天才周一，要不要做上周的？"）
   - 用户表述含糊（如周三说"做个周总结"）→ **必须先问**：

     > "想做哪一周？
     > - 本周（{本周一日期} → 今天）
     > - 上周（{上周一日期} → {上周日日期}）"

   等用户确认后再进入下一步。

3. **读 daily 文件**：按确认的周期，用 Read 工具依次读取每天的 `~/second-brain/daily/YYYY-MM-DD.md`。
   - 某天文件不存在 → 标注"无记录"，**禁止编造**
   - 全周 daily 都不存在 → 告诉用户"这周没有 daily 记录，无法生成周总结"，结束

## 汇总三模块（草稿）

把全周 daily 的三类内容**合并整理**成周视角的表述。要点：

- 来自 daily 已记录的事实，可重组语言、可合并、可概括
- 不写 daily 里没有的事
- 某类整周无内容就写"本周无记录"，不脑补"无"
- 状态保留：完成 = ✅，未完成（仍是 [ ]）= ⏳；不要凭空判断"进行中"

模板：

```markdown
### 💻 一、工作

- ✅ {合并/概括后的事项 1}（{涉及日期或时间点}）
- ✅ {事项 2}
- ⏳ {未完成事项}（{原计划日期}）

### 📚 二、学习

- ✅ {事项 1}
- ✅ {事项 2}

### 🏠 三、生活

- ✅ {事项 1}
- ✅ {事项 2}
```

## 提取关键词（草稿）

基于已汇总的三模块内容，提取 **3-5 个**本周关键词。每个关键词带一句话说明"为什么重要"。

要点：
- 关键词来自实际内容的归纳，不是凭空贴标签
- 说明里不写空话（"很重要" / "有意义"）
- 找不到 3 个就写几个写几个，不凑数

模板：

```markdown
## 🔑 四、本周关键词

1. **{关键词 1}** — {一句话说明为什么重要}
2. **{关键词 2}** — {说明}
3. **{关键词 3}** — {说明}
```

## AI 视角的整体评价（草稿，单独高亮）

**这是编辑模式里最靠近教练的一节，纪律最严**。3-5 句话，内容是基于本周事实做的克制观察。

可以写：
- 观察到的节奏 / 重心 / 模式（基于事实，不是脑补）
- 想抛给用户的疑问（不是结论）
- 值得留意的信号（如某类任务持续未完成、情绪信号等）

不能写：
- 空话套话（"效率很高" / "继续保持" / "本周很充实"）
- 纯赞美 / 纯说教
- 戴帽子的评价（"你应该" / "你需要"）
- 未验证内核就点 patterns 归纳规律（参见 SKILL.md 全局约束第 5 条）

**没有有意义的观察就直接写"本周无新观察"**——参考七条红线第 7 条精神。

模板：

```markdown
## 🤖 五、AI 视角的整体评价

{3-5 句话的观察 / 疑问 / 信号；或"本周无新观察"}
```

## 草稿展示与确认

把上述四节（汇总三模块 + 关键词 + AI 评价）**一次性**贴在对话里给用户审。

要点：

- 以完整 markdown 形式展示
- **AI 视角评价那一节单独再高亮一次**，提示用户："这是 AI 视角的整体评价，比其他模块更主观，请单独确认要不要写入文件。"
- 结尾加："以上周报草稿请用户过目，有需要修改 / 删减 / 替换的地方直接说，确认无误后我写入 reports/。"

用户回应：

- 说"OK / 可以 / 写入吧" → 全部写入
- 指明只删某节（如"AI 评价那段去掉"）→ 写入时跳过该节
- 有修改 → 改完再发一版 → 继续等确认（可多轮）
- **草稿未确认前禁止写文件**

## 写入 reports 文件

文件命名：`~/second-brain/reports/weekly-YYYY-Www.md`（W 后面跟 ISO 周编号，两位数补零，如 `weekly-2026-W21.md`）。

用 Write 工具新建文件，写入完整内容。文件头：

```markdown
# 周总结 {YYYY-Www}

**周期：** {周一日期} → {周日日期}
**生成时间：** {YYYY-MM-DD HH:MM}

---

{上面用户确认后的所有模块}
```

**已存在同名文件时**：先 Read 看一下，问用户是覆盖还是另存为 `weekly-YYYY-Www-v2.md`，**不要静默覆盖**。

## 衔接 +reflect

写入完成后，问：

> "周总结已写入 reports/。要不要 `+reflect --window 7d` 进入深度反思？"

用户回应：

- 说"好 / 反思一下 / +reflect" → 进入 [`reflect.md`](reflect.md) 流程，窗口取本次周总结的周期
- 说"不用 / 算了 / 结束" → 结束，不做 memory 重写（周总结不动 memory，避免和收工 / chat 重复写）

## 完成报告

简短报告给用户：
- 写入的文件路径
- 是否包含 AI 评价段（用户可能要求去掉）
- 是否进入了 reflect（如进入则交给 reflect.md 接管）

## 边界情况

- **a. 全周 daily 都无记录**：直接告诉用户"这周没有 daily 记录，无法生成周总结"，结束。不要"补一份空模板"。
- **b. 周期含糊但用户拒绝确认**：默认按"上周"处理（更常见的场景），并在草稿开头注明"周期默认按上周处理"。
- **c. 用户审稿时说"全删了重做"**：清空草稿，重新进入汇总步骤，可以让用户指定重点关注哪些天 / 哪些事。
- **d. 用户只想要某一节**（如"只要关键词"）：仍生成草稿但只展示用户要的那节，写入也只写那节 + 文件头。
- **e. 跨月周**（如周一在 4 月、周日在 5 月）：文件名按周日所在月份的 ISO 周计算；周期标注完整跨月日期。

## 不做什么（红线提醒）

- 不在草稿未确认前写文件
- 不编造 daily 里没出现的事实（红线 2）
- 不脑补未记录的细节（红线 4）
- 不在 AI 评价里写空话套话或戴帽子
- 不在 AI 评价里未验证就点 patterns 归纳规律
- 不改 profile/（周总结不动 memory，也不提议 patterns/user/soul）
- 不发飞书、不外传 reports/ 内容（隐私边界硬约定）
- 不自动衔接 reflect（必须用户显式同意）

# +checkup — 月度体检

## 语义

每月一次的系统性回顾。比 `+reflect` 更长、更全面，且**是 `patterns.md` 的主更新时机**——一次性整理这个月观察到的所有新 pattern 提议给用户。

v1 不做 cron 自动触发，由用户手动调用。

## 参数

- `--month YYYY-MM`（可选）：要体检的月份。不带参数 = 上个月。
  - 例：今天是 2026-05-11，不带参数 → 体检 2026-04
  - `--month 2026-03` → 体检 2026 年 3 月

## 执行流程

### Step 1 — 解析参数 + 确定月份

- 解析 `--month`，默认上个月（注意年份切换：1 月的"上个月"是上一年的 12 月）
- 计算月份的起止日期

### Step 2 — 加载素材（双源）

- 用 Read 工具读取 `~/second-brain/profile/` 全部 4 个文件（任一文件不存在时跳过，不报错）
- **raw 源（目标月）**：用 Read 工具读取 `~/second-brain/raw/YYYY-MM.md`
- **raw 源（上月对比）**：用 Read 工具读取上一个月的 `raw/YYYY-MM.md`，不存在则跳过
- **daily 源（目标月）**：用 Read 工具读取目标月所有 `~/second-brain/daily/YYYY-MM-DD.md`（用 Bash `ls ~/second-brain/daily/{YYYY-MM}-*.md` 列出）
- **daily 源（上月对比）**：同上但取上一个月所有日记，不存在则跳过
- **最近 checkup**：用 Read 工具读取最近一份 `reports/*-checkup.md`，不存在则跳过

扫描 raw 时，**"套错"维度优先看每段的"分歧 / 我驳回 agent 的地方"栏**——patterns 进化的直接燃料。**跳过标题含 `[元对话]` 的节**——那是协议谈判记录，不属于反思素材。

扫描 daily 时，重点看：
- 反复出现的未完成任务（计划但拖延）
- 计划外完成的事（临时插入背后的动因）
- 描述子 bullet 里的情绪记录（白天没展开的情绪线索）

如果**目标月 raw + daily 都为空** → 立刻告诉用户："这个月没有素材，体检没意义。要不要换一个月（比如 `--month YYYY-MM`）？"，**不要硬挤报告**。

只有一源有内容时正常生成报告，但报告头部"素材"标注说明。

### Step 3 — 生成体检报告

报告路径：`~/second-brain/reports/YYYY-MM-checkup.md`（YYYY-MM 是体检的目标月）。

如果文件已存在（同一月已体检过），加 `-2` 后缀。

**报告必须包含的 5 节**：

```markdown
# 月度体检 · YYYY-MM

> 体检月份：YYYY-MM-01 至 YYYY-MM-末
> 素材：N 段对话 + M 篇日记
> 上次体检：YYYY-MM-checkup.md（或"首次体检"）

## 1. 这个月的画面

> 一段叙事性的总结——这个月用户主要在经历什么？情绪基调？关键事件？
> 不超过 200 字。要让用户读完一节就能"看见这个月"。

## 2. Pattern 演化

### 2.1 已确认的 pattern 这个月怎么表现

> 引用 patterns.md 里现有条目，从四个维度对照：
> - **强化**（多次出现）
> - **淡化**（这个月没出现）
> - **变形**（出现但形态不一样）
> - **套错**（agent 当时点出了这个 pattern，但用户驳回——形似但内核不同）

-

### 2.2 这个月新观察到的候选 pattern（**待用户确认**）

> 这是本节最关键的内容——必须列出来。
> 每条带至少 2 个证据（具体日期 + 一句话）才算"候选"，单次出现不算 pattern。

候选 1：[描述]
- 证据：YYYY-MM-DD（[一句话]）
- 证据：YYYY-MM-DD（[一句话]）
- 建议放进 [显式偏好 / 归纳规律] 分区

候选 2：...

## 3. 和上月对比

> 状态基调 / 关注焦点 / 情绪倾向变了什么？
> 如果是首次体检，本节写"首次体检，无对比"。

-

## 4. 如果只挑一件事改

> 基于这个月所有素材，给用户**一个**最具杠杆的建议——不是 list 5 件事，是 1 件事。
> 要具体、可执行、风险可承受。

[一段话，50-100 字]

## 5. 几个尖锐问题

> 3-5 个，结构和 +reflect 第 4 节相同。
```

### Step 4 — 走 patterns 提议流程

报告写完后，**主动**和用户走一遍报告 2.2 节的候选 pattern：

> "上面 2.2 节列了 N 个候选 pattern，要不要我帮你一条一条过一下，确认哪些写进 patterns.md？"

用户同意 → 一条一条提议，每条等用户回应（同意/拒绝/修改措辞）：
- 同意 → 用 Edit 工具把这一条加到 `patterns.md` 对应分区
- 拒绝 → 不写，但**也把"已讨论过、未通过"标注到本次 checkup 报告里**（避免下次又冒出来）
- 修改 → 按用户修改后的版本写

用户拒绝整体过一遍 → 不写 patterns，候选清单留在报告里供用户日后自己看。

### Step 4.5 — 回填反例 / 边界（对应 2.1 节"套错"维度）

对 2.1 节里被标"套错"的每条已有 pattern：

- 口头提议：
  > "这个月 X 这条 pattern 被套错过（YYYY-MM-DD 那次），要不要给它加一条反例？反例可以写成：[基于驳回情境的一句话]"
- 用户同意 → Edit 工具在 `patterns.md` 对应条目下加 `反例 / 边界：xxx`
- 用户拒绝 → 不写，但把"已讨论过、未通过"标到本次 checkup 报告里
- 用户改措辞 → 按修改后的版本写

**判定：某条 pattern 的反例 ≥ 2 条时**，不再加第 3 条反例，改为提议**重写这条 pattern 本身**——措辞过粗才会反复套错。重写仍走"提议-确认"流程，不擅改。

### Step 5 — 完成后

向用户简短报告：
- 报告路径
- 这个月被加入 patterns.md 的条目数（如果有）
- 不剧透报告内容，让用户自己看

## 边界情况

- **目标月 raw + daily 都为空**：见 Step 2。
- **目标月素材极少**（< 3 段对话 + 日记合计）：照常生成报告，但每节相应简短。2.2 节的候选 pattern 要诚实——素材不够就明确说"本月素材不足以观察 pattern"。
- **同一月重复体检**：允许，加 `-2` 后缀。提醒用户"这个月已体检过，新报告可能和上次重叠"。
- **首次体检**：第 3 节写"首次体检，无对比"。第 2.1 节如果 patterns.md 还是空模板，就跳过。
- **只有 daily 没有 raw**：合法。"Pattern 演化"和"尖锐问题"内容会更偏事实层，这是可接受的——但要在报告里诚实说明本月没有 +chat 素材。

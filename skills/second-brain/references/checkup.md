# +checkup — 月度回顾

## 语义

每月一次的系统性回顾——融合了原 +reflect 的跨时间观察能力（主题聚类、矛盾点）。素材源：raw（教练对话）+ daily（日记）双源 + 上月对比 + 最近一次 checkup 报告。

**报告本身是核心交付物**。patterns 维护从「主动主线」降为「opt-in 副产品」——但其中具有「健康告警」性质的（累计反例 ≥ 2、连续淡化 ≥ 3 月、多次驳回）由 agent 主动提议，避免规则失效却无人维护。

v1 不做 cron 自动触发，由用户手动调用。

## 参数

- `--month YYYY-MM`（可选）：要回顾的月份。不带参数 = 上个月。
  - 例：今天是 2026-05-11，不带参数 → 回顾 2026-04
  - `--month 2026-03` → 回顾 2026 年 3 月

## 执行流程

### Step 1 — 解析参数 + 确定月份

- 解析 `--month`，默认上个月（注意年份切换：1 月的"上个月"是上一年的 12 月）
- 计算月份的起止日期

### Step 2 — 加载素材（双源）

- 用 Read 工具读取 `~/second-brain/profile/` 全部 4 个文件（任一文件不存在时跳过，不报错）
- 用 Read 工具读取 `~/second-brain/profile/patterns-archive.md`（不存在则跳过——首次归档时才会建）
- **raw 源（目标月）**：用 Read 工具读取 `~/second-brain/raw/YYYY-MM.md`
- **raw 源（上月对比）**：用 Read 工具读取上一个月的 `raw/YYYY-MM.md`，不存在则跳过
- **daily 源（目标月）**：用 Read 工具读取目标月所有 `~/second-brain/daily/YYYY-MM-DD.md`（用 Bash `ls ~/second-brain/daily/{YYYY-MM}-*.md` 列出）
- **daily 源（上月对比）**：同上但取上一个月所有日记，不存在则跳过
- **最近 checkup**：用 Read 工具读取最近一份 `reports/*-checkup.md`，不存在则跳过

扫描 raw 时，**"套错"维度优先看每段的"分歧 / 我驳回 agent 的地方"栏**——patterns 进化的直接燃料。**跳过标题含 `[元对话]` 的节**——那是协议谈判记录，不属于反思素材。

扫描 daily 时，重点看：
- 反复出现的未完成任务（计划但拖延）
- 计划外完成的事（临时插入背后的动因）
- 描述子 bullet 里的情绪记录（白天没展开的情绪线索）

如果**目标月 raw + daily 都为空** → 立刻告诉用户："这个月没有素材，回顾没意义。要不要换一个月（比如 `--month YYYY-MM`）？"，**不要硬挤报告**。

只有一源有内容时正常生成报告，但报告头部"素材"标注说明。

### Step 3 — 生成回顾报告

报告路径：`~/second-brain/reports/YYYY-MM-checkup.md`（YYYY-MM 是回顾的目标月）。

如果文件已存在（同一月已回顾过），加 `-2` 后缀。

**报告必须包含的 7 节**：

```markdown
# 月度回顾 · YYYY-MM

> 回顾月份：YYYY-MM-01 至 YYYY-MM-末
> 素材：N 段对话 + M 篇日记
> 上次回顾：YYYY-MM-checkup.md（或"首次回顾"）

## 1. 这个月的画面

> 一段叙事性的总结——这个月用户主要在经历什么？情绪基调？关键事件？
> 不超过 200 字。要让用户读完一节就能"看见这个月"。

## 2. 反复出现的主题

> 把月内 raw + daily 里反复出现的事情聚类——哪些事情其实是同一件事的不同切面？
> 给每个主题一个名字 + 一段说明 + 引用具体日期作证据。
> 单次出现不算主题。

### 主题 A：[名字]

[说明]

证据：YYYY-MM-DD（[一句话回忆]）、YYYY-MM-DD（...）

### 主题 B：[名字]

...

## 3. 矛盾点

> 用户在不同时点说过的相互矛盾或相互拉扯的话。
> 不是要批评，是要让用户自己看到。

- 你 YYYY-MM-DD 说 "X"，但 YYYY-MM-DD 又说 "Y"——这两个怎么共存？
-

## 4. 和上月对比

> 状态基调 / 关注焦点 / 情绪倾向变了什么？
> 如果是首次回顾，本节写"首次回顾，无对比"。

-

## 5. Pattern 演化

### 5.1 已确认的 pattern 这个月怎么表现

> 引用 patterns.md 里现有条目，从四个维度对照：
> - **强化**（多次出现）
> - **淡化**（这个月没出现；连续淡化 3 个月以上的进 5.3 健康告警）
> - **变形**（出现但形态不一样）
> - **套错**（agent 当时点出了这个 pattern，但用户驳回——形似但内核不同）

-

### 5.2 这个月新观察到的候选 pattern（**Step 6 opt-in 时再过审**）

> 每条带至少 2 个证据（具体日期 + 一句话）才算"候选"，单次出现不算 pattern。

候选 1：[描述]
- 证据：YYYY-MM-DD（[一句话]）
- 证据：YYYY-MM-DD（[一句话]）
- 建议放进 [显式偏好 / 归纳规律] 分区

候选 2：...

### 5.3 健康告警（**Step 5 由 agent 主动讨论**）

> 这一节列出 pattern 库里**已经亮红灯**的条目，agent 必须主动和用户讨论。
> 仅在以下情况入库：
> - 累计反例 ≥ 2（含本月新增 / 历史已有）——措辞过粗，建议重写或归档
> - 连续 3 个月以上"淡化"（5.1 维度）——建议归档
> - 已被用户多次显式驳回——建议归档
> 没有告警时本节写"无"。

- [pattern 名] — 告警类型：累计反例 N 条 / 连续淡化 N 个月 / 多次驳回
  - 建议动作：[重写 / 归档]
  - 理由：[基于本月观察的一句话]

## 6. 如果只挑一件事改

> 基于这个月所有素材，给用户**一个**最具杠杆的建议——不是 list 5 件事，是 1 件事。
> 要具体、可执行、风险可承受。

[一段话，50-100 字]

## 7. 几个尖锐问题

> 3-5 个反问。要具体、要锋利、要基于上面的观察。
> 不要泛泛的"你怎么看？"

1.
2.
3.
```

### Step 4 — 交付报告

报告写入后，**立刻**向用户简短交付：
- 报告路径
- 主要发现一两句话（不剧透具体反问和告警细节）

> "本月回顾已写入 reports/YYYY-MM-checkup.md。主要画面是 [一句话]。报告里有 N 个候选 pattern、M 条健康告警，要不要现在过一下？"

**到这里 checkup 的核心交付已完成**。后面的 Step 5、Step 6 都是可拒绝的追加项。

### Step 5 — 健康告警讨论（agent 主动，每条必提）

仅当报告 **5.3 节非空** 时执行。

对 5.3 节每条告警，agent **必须主动**和用户走一遍——这是规则失效的早期信号，不能 opt-in。但每条仍由用户决定动作：

**累计反例 ≥ 2**：
> "X 这条 pattern 累计已有 N 次反例。措辞可能过粗——要不要重写，或者归档？"

- 用户选**重写** → agent 提议新措辞 → 用户确认 → 用 Edit 工具改 `patterns.md` + 把旧版本追加到 `patterns-archive.md`（带归档原因 + 日期）
- 用户选**归档** → 从 `patterns.md` 删除该条 + 追加到 `patterns-archive.md`
- 用户选**保留** → 不动，但在本次 checkup 报告里标"已讨论过、保留"（避免下次又被告警）

**连续淡化 ≥ 3 月**：
> "X 这条 pattern 已经连续 N 个月没在对话里出现。要不要归档？以后用得上随时从 archive 翻出来。"

- 流程同上（重写选项可省，主要是归档/保留二选一）

**多次驳回**：
> "X 这条 pattern 这个月被你显式驳回 N 次。要不要归档或重写？"

- 流程同上

讨论完所有告警后，agent 总结："健康告警处理完毕——归档 N 条 / 重写 M 条 / 保留 K 条。"

### Step 6 — 候选 pattern 入库 + 单次套错回填（opt-in）

仅当报告 **5.2 节非空 或 5.1 节有未触发告警的轻量套错** 时执行。

agent 问一次：

> "5.2 节列了 N 个候选 pattern，5.1 节里还有 M 个轻量套错（反例还没到 2 次）。要不要现在一起过一下？不过也行，候选清单留在报告里，下次想看随时翻。"

**用户同意**：

- 候选 pattern 一条一条过，每条等用户回应：
  - 同意 → 用 Edit 工具加到 `patterns.md` 对应分区
  - 拒绝 → 不写，把"已讨论过、未通过"标到本次 checkup 报告里
  - 修改 → 按用户修改后的版本写
- 轻量套错对每条提议反例措辞 → 用户确认后 Edit 工具加到 `patterns.md` 对应条目下 `反例 / 边界：xxx`

**用户拒绝**：候选清单和未回填反例都留在报告里，不写 patterns。

### Step 7 — 最终完成报告

向用户简短报告：
- 报告路径
- 本月动作汇总：归档 N 条 / 重写 M 条 / 新增 K 条 / 反例回填 L 条（用户拒绝 Step 6 时这些都是 0）
- 不剧透报告主要内容，让用户自己看

## 边界情况

- **目标月 raw + daily 都为空**：见 Step 2。
- **目标月素材极少**（< 3 段对话 + 日记合计）：照常生成报告，但每节相应简短。5.2 节的候选 pattern 要诚实——素材不够就明确说"本月素材不足以观察 pattern"。
- **同一月重复回顾**：允许，加 `-2` 后缀。提醒用户"这个月已回顾过，新报告可能和上次重叠"。
- **首次回顾**：第 4 节写"首次回顾，无对比"。第 5.1 节如果 patterns.md 还是空模板，就跳过；5.3 健康告警节也写"无"（首次没有历史月份判断淡化）。
- **只有 daily 没有 raw**：合法。"反复出现的主题"和"矛盾点"内容会更偏事实层。这是可接受的——但要在报告里诚实说明本月没有 +chat 素材。
- **`patterns-archive.md` 不存在**：第一次需要归档时由 agent 用 Write 工具创建，文件首行 `# Patterns 归档`，再追加首条。后续用 Read → Write 覆写模式追加（不要用 Edit 追加，月度文件追加场景下不可靠）。
- **判断"连续淡化 N 月"的实现**：扫描最近 N 个月的 `reports/*-checkup.md`，看该 pattern 是否每次都在 5.1 节"淡化"维度里。没有历史 checkup 时不触发淡化告警。

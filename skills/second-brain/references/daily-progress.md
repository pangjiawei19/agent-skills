# 完成了 xxx — daily 段进度更新

## 语义

用户说"完成了 xxx" / "搞定了 xxx"= 把今日清单里对应条目打勾、追加完成时间。Agent 仍是秘书模式。

## 启动流程

1. **加载 profile**（每个 shortcut 调用都加载，不省略；任一文件不存在时跳过，不报错）：
   - `~/second-brain/profile/user.md`
   - `~/second-brain/profile/soul.md`
   - `~/second-brain/profile/patterns.md`
   - `~/second-brain/profile/memory.md`
2. **读今日 daily 文件**：`~/second-brain/daily/YYYY-MM-DD.md`
   - 不存在 → 走"daily 文件不存在"分支（见边界情况 h）

## 打勾流程

1. 在文件里**找对应条目**：
   - 字面一致或语义合理匹配（如清单写"EKS MCP 接口替换"、用户说"完成了 EKS 接口" → 匹配）
   - 用户描述和清单项明显对不上任何一项 → 计划外，追加到对应大类末尾
2. **改写**：
   - `- [ ] (TODO) 描述` → `- [x] (HH:MM) 描述`（只换括号内容，描述原封不动）
   - HH:MM 用 24 小时制，以收到用户消息的时刻为准
3. **原描述一个字都不动**（daily 红线 3）。如果用户顺带讲了细节，用子 bullet 追加：
   ```markdown
   - [x] (10:23) EKS MCP 接口替换
     - 联调通过，延迟 200ms → 80ms
   ```
4. 使用 Edit 工具（清单条目是唯一锚点，可安全 Edit）

## 计划外任务

用户讲的事不在今日清单 → 直接追加为已完成态：

```markdown
### 💻 工作
- [ ] (TODO) {原有未完成项}
- [x] (HH:MM) {新讲的事}
```

按任务性质归入对应分区（工作 / 学习 / 生活），不是默认追加到工作。

**不**做"临时"或"计划外"标记——今天完成的就是今天完成的（daily 红线对应）。

## 情绪 / 想法夹带

用户可能在任务消息里夹带情绪（例："搞定了 EKS，但被老板那句话搞得很烦"）。

**秘书模式不接情绪话头**——
- 打勾时把情绪原话作为子 bullet 照抄记录：
  ```markdown
  - [x] (10:23) EKS MCP 接口替换
    - 被老板那句话搞得很烦
  ```
- **不**反问、**不**挑战、**不**点 pattern
- 这部分会在收工切教练后浮上来（agent 看 daily 日记发现"白天有情绪记录但没展开"，作为承接素材）

## 元对话识别

用户白天可能讲一些**不是任务、而是关于 agent 行为本身或 skill 协议**的话——典型信号：

- "你以后开工别问那么多废话"
- "完成时别追加时间戳，那个时间我自己看得到"
- "你刚才那个反问太烦了"
- 涉及 `soul.md` / `patterns.md` / `user.md` 或本 skill 协议本身

**秘书模式不处理这些**——不当场改 patterns/soul/user、不反问、不挑战、不指出"这是元对话"。

处理方式：

1. 在当日 daily 文件追加 `## 💬 待讨论` 区（区不存在则创建，位置放在 `## 📋 今日任务清单` 之后、复盘段之前），追加一条：

   ```markdown
   ## 💬 待讨论

   - (HH:MM) 用户原话照抄
   ```

2. 口头回应固定一句："记着了，收工时一起过。"——不展开、不反问、不承诺改什么。

3. 用 Edit 工具追加；区已存在则在区末尾追加新条目，不覆盖。

**不判断**用户讲的是"真要改协议"还是"单纯吐槽"——秘书一律记，由收工/教练段决定哪条要落 profile。

**信号边界**：纯情绪话（如"被老板搞烦了"）不是元对话，走"情绪 / 想法夹带"流程（记进对应任务的子 bullet）。元对话核心特征是**指向 agent 行为或 skill 协议**。

## 边界情况

- **a. 用户说"完成了 xxx"但 xxx 在清单里是子项细节而非条目本身**：**先不打勾父条目**，只在父条目下追加子 bullet `- (HH:MM) 完成了 xxx 部分`。然后询问用户"父条目要整条打勾吗？"——用户确认才打勾父条目
- **b. 同一条目用户说"完成了"两次**：第二次告诉用户"这条已经打过勾了（HH:MM）"，不重复改写
- **c. 用户说"完成了 xxx"但 xxx 描述太模糊（如"完成了那个事"）**：反问一句具体是哪条，**不**猜——猜错就违反红线 4
- **d. 用户说"撤销 xxx 的完成状态"**：把 `- [x] (HH:MM) 描述` 改回 `- [ ] (TODO) 描述`（只换括号内容）
- **h. 今日 daily 文件不存在**（用户跳过了开工）：agent 主动询问"今天还没开工，要不要先列清单？还是直接把这件事记为今天完成的？"。用户说"直接记" → 创建 daily 文件，把该项作为计划外 `- [x] (HH:MM) 描述` 直接追加，按三类归到合适分区（agent 自行判断分类，明显错的用户会纠）

## 不做什么（红线提醒）

- 不脑补"应该完成得不错"这种评价
- 不扩写用户没讲的细节
- 不另起"已完成"段落把这条挪过去（daily 红线 1）
